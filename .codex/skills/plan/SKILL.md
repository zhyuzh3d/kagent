# Plan

## 目标
在不改动业务实现的前提下，结合可执行分析结果产出高质量、可执行的项目规划、设计方案，并在需要时落盘为 `*.md` 文档（规划类文档默认写入 `plan/`）。

## 触发规则
1. 仅当用户消息中存在“明确模式指令” `plan` 时触发，且 `plan` 必须是独立标记：
   - `plan` 前一字符为消息开头，或为空白，或为中文/英文标点（如 `，,:：；。!?`）。
   - `plan` 后一字符为消息结尾，或为空白，或为中文/英文标点（如 `，,:：；。!?`）。
   - 特别地：`plan` 后一字符若为 `.` `/` `\\` `-` `_` 或字母/数字，视为“仍在标识符/路径/文件名中”，不算模式指令（例如 `plan.md`、`/plan`、`planning`、`plan_v2`）。
2. 允许形式（可在句首或句中）：`plan`、`plan，...`、`plan: ...`、`plan：...`、`... plan，...`。
3. 不允许形式（避免误触发）：`planning`、`plan.md`、`/plan`、代码块/反引号里的 `plan`、URL 中的 `plan`。
4. 若同一条消息里同时出现多个模式指令，按最高权限执行：`dev` > `plan` > `chat`。出现 `dev` 时，本 Skill 不应阻止编码与文件操作（仍可同时产出文档）。

## 执行范围
1. 需求澄清与功能边界定义。
2. 技术方案对比与架构草案。
3. 开发计划、里程碑、任务拆解、排期建议。
4. 验收标准、测试策略、风险清单、回滚预案。
5. 其他纯文档设计工作（PRD、技术设计、实施计划等）。
6. 允许对项目内任意路径下的 `*.md` 文档执行创建、修改、重命名、删除：
   - 规划类文档默认写入 `plan/`（除非用户明确指定路径）。
   - 默认命名遵循 `AGENTS.md` 的 `plan/` 约定：`plan/<yymmdd>-<主题>-<类型>.md`（不含时分秒，类型后缀为 `-prd/-devplan/-devreport/-ana`）。
   - 项目说明与开发日志等项目文档可写入 `doc/`（例如 `doc/_instruction.md`、`doc/_devlog.md`）。
7. 允许在项目内生成、更新、删除临时分析产物、缓存、中间文件或报告文件。
8. 对非 Markdown 的现有项目文件，仅允许分析读取，不得修改、删除、重命名。

## 执行步骤
1. 先验证本条消息是否“明确呼唤 plan 模式”；若不是（例如只是出现子串或出现在代码/文件名中），立即停止并不应用本 Skill。
2. 去除模式指令 `plan`（以及紧随其后的 `:`/`：`/`,`/`，` 等标点），识别用户目标与约束，补齐关键假设。
3. 读取必要上下文（相关文档与代码结构）以保证方案贴合真实项目；默认优先读 `doc/_instruction.md` 和 `AGENTS.md`。
4. 为保证方案真实可执行，可按需执行分析命令、读取项目与本地可读文件、浏览网页、查看图片/PDF、查询 Git 历史，或调用会在项目内生成临时文件的分析工具；不得修改、删除、重命名非 Markdown 的现有项目文件。
5. 输出结构化文档：背景、目标、范围、方案、任务分解、风险、验收标准。
6. 若用户要求落盘，则创建或更新对应 `*.md` 文件；规划类文档默认写入 `plan/`（除非用户明确指定路径）。若只是分析需要，也可在项目内创建和清理临时分析产物。
   - 当默认写入 `plan/` 且用户未指定文件名时，必须使用 `yymmdd-` 日期前缀（不含时分秒），并按文档类型追加后缀：`-prd/-devplan/-devreport/-ana`。
7. 若更新 `doc/_devlog.md`，必须遵守“只追加不改历史记录”；若更新 `doc/_instruction.md`，必须以仓库真实结构与可核验事实为准。
8. 若用户要求删除或重命名文档文件，仅允许操作 `*.md` 目标。
9. 若同一条消息里同时出现 `dev`，按最高权限进入 `dev` 模式，本 Skill 不应阻止编码与文件操作（仍可同时产出文档）。
10. 若用户请求“直接开发”或要求改写非 Markdown 的现有项目文件但未呼唤 `dev`，先给可执行计划或分析结果，并提示用户再次明确呼唤 `dev` 才进入开发执行。

## 硬性边界
1. 不执行编码实现，不修改、删除、重命名非 Markdown 的现有项目文件。
2. 允许在项目内创建和清理临时分析产物，但不得借此覆盖任何现有项目文件。
3. 不执行提交代码、部署发布、破坏性清理等超出分析与规划范围的动作。
4. 不宣称“已实现”“已修复”或“已落盘”任何未实际发生的内容。

## 完成标准
1. 交付的规划/设计文档结构完整、可直接执行。
2. 文档明确范围、任务、风险和验收条件。
3. 本轮若发生正式文档写入、删除或重命名，目标必须全部为 `*.md` 文档；规划类文档默认写入 `plan/`；若产生其他文件，仅限临时分析产物，且未发生实际编码实现。

## 权限持续性
`plan` 默认是一条消息内的一次性授权。下一条消息若未再次明确呼唤 `plan` 或 `dev`，默认回到 `chat` 权限。
