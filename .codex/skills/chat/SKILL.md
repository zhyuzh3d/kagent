# Chat

## 目标
将本轮请求限制为分析讨论型对话协作，基于项目上下文和分析结果提供解释、判断、建议与风险结论，在保证严谨准确的前提下默认输出更精简、更聚焦，并在需要时先核实关键信息再回答。

## 触发规则
1. 仅当用户消息中存在“明确模式指令” `chat` 时触发，且 `chat` 必须是独立标记：
   - `chat` 前一字符为消息开头，或为空白，或为中文/英文标点（如 `，,:：；。!?`）。
   - `chat` 后一字符为消息结尾，或为空白，或为中文/英文标点（如 `，,:：；。!?`）。
   - 特别地：`chat` 后一字符若为 `.` `/` `\\` `-` `_` 或字母/数字，视为“仍在标识符/路径/文件名中”，不算模式指令（例如 `chatgpt`、`/chat`、`chat.md`、`chat_v2`）。
2. 允许形式（可在句首或句中）：`chat`、`chat，...`、`chat: ...`、`chat：...`、`... chat，...`。
3. 不允许形式（避免误触发）：`chatgpt`、`wechat`、`/chat`、`chat.md`、代码块/反引号里的 `chat`、URL 中的 `chat`。
4. 若同一条消息里同时出现多个模式指令，按最高权限执行：`dev` > `plan` > `chat`。出现 `dev` 时，本 Skill 不应施加任何“只对话”限制。

## 执行步骤
1. 先验证本条消息是否“明确呼唤 chat 模式”；若不是（例如只是出现子串或出现在代码/文件名中），立即停止并不应用本 Skill。
2. 去除模式指令 `chat`（以及紧随其后的 `:`/`：`/`,`/`，` 等标点），提取用户真实问题。
3. 为保证真实性，可按需执行分析命令、读取项目与本地可读文件、浏览网页、查看图片/PDF、查询 Git 历史，或调用会在项目内生成临时文件的分析工具；前提是不修改、删除、重命名任何现有项目文件。
4. 允许在项目内创建、更新、删除临时分析产物、缓存、中间文件或报告文件，但应避免与现有项目文件混淆。
5. 若存在关键不确定点或高风险结论，先做最小必要核实：
   - 优先核实本地事实（代码/配置/文档/Git 历史/可复现命令输出）。
   - 在环境允许且有必要时，再做网络检索交叉验证。
   - 无法核实时，必须显式标注假设与不确定性。
6. 按“精简优先”的结构输出：先结论，再给必要的精简的关键依据或约束，除非用户明确要求，不做面面俱到的长清单罗列。
7. 若用户请求实际开发动作（改现有文件、落地实现），只给出建议与计划，不执行这些变更；并提示用户用明确模式指令 `dev` 来启动开发执行。

## 输出规范（chat 默认）
1. 聚焦主题：只回答与当前对话上下文主问题直接相关的内容；不扩展到无关面。
2. 精简表达：默认以短段落为主，避免大段条目罗列；需要列点时，优先“高价值关键点”。
3. 严谨准确：能核实的优先核实；不能核实的明确说明“未核实/待确认”，不要用猜测替代事实。
4. 深度优先：少说结论外的话，多给关键推理、关键权衡与决定性证据来源位置（例如具体文件/命令/日志片段）。

## 权限持续性
`chat` 是默认权限。即使上一条消息使用过 `plan` 或 `dev`，若本条未明确呼唤 `plan` 或 `dev`，也按 `chat` 权限处理。

## 硬性边界
1. 不修改、删除、重命名任何现有项目文件。
2. 允许在项目内创建和清理临时分析产物，但不得借此覆盖现有文件。
3. 不执行提交代码、部署发布、安装依赖后落地实现、破坏性清理等超出分析范围的动作。
4. 若某个命令可能触达现有项目文件且无法确认安全性，则不执行。

## 完成标准
1. 已提供清晰对话答复或建议。
2. 本轮未修改、删除、重命名任何现有项目文件；若产生文件，仅限临时分析产物。
3. 当用户请求落地实现或改写现有文件时，明确提示切换到 `dev` 模式。
