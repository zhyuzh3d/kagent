# Chat

## 目标
将本轮请求限制为纯对话协作，提供可执行建议但不落地代码操作。

## 触发规则
1. 仅当用户消息中存在“明确模式指令” `chat` 时触发，且 `chat` 必须是独立标记：
   - `chat` 前一字符为消息开头，或为空白，或为中文/英文标点（如 `，,:：；。!?`）。
   - `chat` 后一字符为消息结尾，或为空白，或为中文/英文标点（如 `，,:：；。!?`）。
   - 特别地：`chat` 后一字符若为 `.` `/` `\\` `-` `_` 或字母/数字，视为“仍在标识符/路径/文件名中”，不算模式指令（例如 `chatgpt`、`/chat`、`chat.md`、`chat_v2`）。
2. 允许形式（可在句首或句中）：`chat`、`chat，...`、`chat: ...`、`chat：...`、`... chat，...`。
3. 不允许形式（避免误触发）：`chatgpt`、`wechat`、`/chat`、`chat.md`、代码块/反引号里的 `chat`、URL 中的 `chat`。
4. 若同一条消息里同时出现多个模式指令，按最高权限执行：`dev` > `plan` > `chat`。出现 `dev` 时，本 Skill 不应施加任何“只对话”限制。

## 执行步骤
1. 先验证本条消息是否“明确呼唤 chat 模式”；若不是（例如只是出现子串或出现在代码/文件名中），立即停止并不应用本 Skill。
2. 去除模式指令 `chat`（以及紧随其后的 `:`/`：`/`,`/`，` 等标点），提取用户真实问题。
3. 仅输出对话内容：解释、分析、方案比较、风险提示、建议步骤。
4. 若用户请求实际开发动作（改代码、运行命令、改文件），只给出建议与计划，不执行任何操作；并提示用户用明确模式指令 `dev` 来启动开发执行。

## 权限持续性
`chat` 是默认权限。即使上一条消息使用过 `plan` 或 `dev`，若本条未明确呼唤 `plan` 或 `dev`，也按 `chat` 权限处理。

## 硬性边界
1. 不执行终端命令。
2. 不修改、创建、删除任何文件。
3. 不运行测试、构建、安装依赖。
4. 不提交代码或变更仓库状态。

## 完成标准
1. 已提供清晰对话答复或建议。
2. 本轮无任何代码与文件实际变更。
3. 当用户请求落地实现时，明确提示切换到 `dev` 模式。
