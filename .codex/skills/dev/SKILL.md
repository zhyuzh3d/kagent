# Dev

## 目标
在确认需求后执行真实开发工作，并产出可验证结果。

## 触发规则
1. 仅当用户消息中存在“明确模式指令” `dev` 时触发，且 `dev` 必须是独立标记：
   - `dev` 前一字符为消息开头，或为空白，或为中文/英文标点（如 `，,:：；。!?`）。
   - `dev` 后一字符为消息结尾，或为空白，或为中文/英文标点（如 `，,:：；。!?`）。
   - 特别地：`dev` 后一字符若为 `.` `/` `\\` `-` `_` 或字母/数字，视为“仍在标识符/路径/文件名中”，不算模式指令（例如 `dev.go`、`/dev`、`dev_test`）。
2. 允许形式（可在句首或句中）：`dev`、`dev，...`、`dev: ...`、`dev：...`、`... dev，...`。
3. 不允许形式（避免误触发）：`dev.go`、`dev.md`、`device`、`developer`、代码块/反引号里的 `dev`、路径或命令片段里的 `dev`。
4. 若同一条消息里同时出现多个模式指令，按最高权限执行：`dev` > `plan` > `chat`。出现 `dev` 时允许同时产出文档与代码改动。

## 执行步骤
1. 先验证本条消息是否“明确呼唤 dev 模式”；若不是（例如只是出现子串或出现在代码/文件名中），立即停止并不应用本 Skill。
2. 去除模式指令 `dev`（以及紧随其后的 `:`/`：`/`,`/`，` 等标点），解析用户真实需求与约束。
3. 优先读取项目关键文档：`AGENTS.md`、`doc/_instruction.md`（必要时回退 `Instruction.md`）、`doc/_devlog.md`（需要历史时）。
4. 扫描并定位相关代码/配置文件，制定最小可行改动。
5. 实施修改：编码、重构、修复、配置调整、测试补充（按需求）。
6. 执行必要验证：至少进行与改动直接相关的检查（测试/构建/静态检查）。
7. 输出结果：说明改动文件、关键逻辑、验证结果、风险与后续建议。
8. 文档落盘约定（与需求文档同名）：
   - 当用户意图为 `制定开发计划`：在进入开发实现前，必须将开发计划写入 `plan/<需求文档基名>-dev-plan.md`。
   - 当用户意图为 `评估开发结果`：在开发结束后，必须基于最终代码变更与过程问题/经验，整合撰写并写入 `plan/<需求文档基名>-dev-result.md`。
   - 若用户未明确给出需求文档路径，优先使用“最近一次明确提到的需求文档”；无法判定时必须追问，避免写错文件名。

## 行为边界
1. 允许执行代码与文件变更操作。
2. 严禁编造“已完成”结果；无法验证时必须明确说明未验证项。
3. 非用户要求情况下，不做无关重构和大规模风格化改动。
4. 遇到高风险或破坏性动作时，先征得用户确认（例如大规模删除文件、不可逆重写、明显可能导致数据丢失/服务中断的操作）。

## 权限持续性
`dev` 默认是一条消息内的一次性授权。下一条消息若未再次明确呼唤 `dev` 或 `plan`，默认回到 `chat` 权限。

## 完成标准
1. 用户要求的开发任务已实际落地到项目文件。
2. 关键验证步骤已执行，或明确说明无法执行的原因。
3. 结果说明可追溯到具体文件与改动内容。
