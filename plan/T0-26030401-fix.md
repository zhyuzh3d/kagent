# 需求文档：T0-26030401-fix (解决对话流水线的延迟与TTS中断问题)

## 1. 背景与问题描述
在 T0 MVP 版本中，当前对话流水线（LLM增量 -> TTS分段）存在以下严重体验问题：
1. **TTS中途丢失与不稳定**：多句 LLM 响应时，后面的句子极易漏读，甚至偶尔完全不发声（报错：WS Connection/Session limits or timeouts）。
2. **LLM 流式延迟显示**：前端等待长达 2-5 秒才看到第一段文字，流式体验不明显。
3. **分句朗读等待严重**：前端文字已大段显示，但 TTS 由于内置了固定字符数截断（16字）并频繁起停 WebSocket，导致首句朗读启动太慢，且句间存在非自然停顿。

## 2. 根因分析
1. **并发/频发建连冲突**：当前 `TTSClient.Synthesize` 是单次请求即断开 WS 的模式，在单个回合中（Turn）为每一句话重复建立 WS 连接、StartSession、TaskRequest、FinishSession，消耗极大，且易触发服务端的并发/频控限制。
2. **LLM 到 TTS 的缓冲逻辑低效**：`internal/pipeline.go` 内的分段缓冲死等 16 字符，加上非持续流式的 TTS 调用方式，导致延误堆积。
3. **第一字节时间 (TTFB) 处理空白**：未将大模型首个 chunk 到达这一个明确事件抛出给前端，甚至把状态切换留到了最后。

## 3. 需求目标
1. **单回合单连 TTS**：为每个 Turn 建立唯一的 TTS WebSocket 流连接，保持长链接（真正使用 Bidirectional 流式协议），随着大模型的流式字串边转边发，Turn 结束时再断开。
2. **聪明的标点断句**：调整或废除单纯基于固定字数的分段，改为基于语意停顿标点（，。！？等）的增量触发，降低首段文字发声的延迟。
3. **即时状态反馈**：当 LLM 回调第一个 delta 数据时，确保前端理解系统正在积极响应。

## 4. 预期验收标准
- [ ] 流畅对话：用户长篇提问或者大模型大段回复时，语音完整播报，不截断、不漏句。
- [ ] 极低延迟：LLM 首段文字显示后，语音合成马上跟进启动播报（<1秒感知延迟）。
- [ ] 稳定后台：不出现 `websocket: bad handshake` 或 `TTS connection failed` 高频错误。
