# T0-2026-03-03-01 MVP 开发计划（分段近实时 + 真实接口直连）

## 简要摘要
1. 实施策略已锁定：先做“分段近实时”闭环（ASR 连续上行、LLM 增量、按短句触发 TTS 分段播放），保证稳定可用与可打断；后续再升级链路级全流式。  
2. 协议不确定性处理已锁定：采用 Provider 适配层 + 真实接口直连，协议字段差异只在适配层修正，不污染业务状态机。  
3. 对需求文档的必要调整：Go 标准库不提供完整 WebSocket 服务端/客户端能力，允许引入 1 个必要依赖 `github.com/gorilla/websocket`，其余保持原生极简。  
4. 本轮交付目标：本地 `127.0.0.1` 单进程可运行，具备 Start/Stop、ASR 文本、LLM 文本、TTS 播放、barge-in、错误可见、资源可回收。  

## 目标与范围
1. In Scope：单用户单页面会话；端到端语音对话；打断；可观测状态；本地运行。  
2. Out of Scope：数据库、记忆系统、人格、多人房间、公网部署。  
3. 验收基线：满足 `plan/T0-2026-03-03-01.md` 的 10.1 功能 DoD 与 10.3 手工测试清单。  

## 关键接口与类型（公共行为变更）
1. WebSocket 上行控制消息（文本 JSON）：`start`、`stop`、`interrupt`、`utterance_end`。  
2. WebSocket 下行事件（文本 JSON）：`status`、`error`、`asr_partial`、`asr_final`、`llm_delta`、`llm_final`、`tts_chunk`。  
3. WebSocket 下行音频（二进制）：遵循“先 `tts_chunk` 事件后 binary”的配对协议；`tts_chunk` 必含 `turn_id` + `seq` + `format`。  
4. 核心状态机：`Idle -> Connecting -> Listening -> Recognizing -> Thinking -> Speaking -> Interrupted/Error -> Listening/Idle`。  
5. `turn_id` 规则：后端单调递增 `uint64`，所有文本事件和音频分片均绑定同一 `turn_id`。

## 实施方案（决策完备）
1. 服务层：`main.go` 启动 HTTP 与 `/ws`，静态资源挂载 `/webui/`（兼容 `/static/`），监听 `127.0.0.1:18080`。  
2. 协议层：`protocol.go` 定义消息结构、事件编码、`tts_chunk` 配对规则，统一 `ts_ms` 生成。  
3. 会话层：`session.go` 维护连接、状态机、`turn_id`、取消函数、队列与背压。  
4. ASR 层：`asr.go` 负责与 `asr_s` 流式连接，输入 PCM16LE 16k mono 分片，输出 partial/final/endpoint 事件。  
5. LLM 层：`llm.go` 负责流式 `chat`，输出 delta/final；接收取消信号立即终止请求。  
6. TTS 层：`tts.go` 以“短句分段”调用 `tts_s`，每段返回后生成 `tts_chunk` + binary 下行。  
7. 编排层：`pipeline.go` 串联 ASR -> LLM -> TTS；实现 turn 生命周期、错误传播、stop/interrupt 清理。  
8. 前端采集：`index.html` 采用 `getUserMedia` + `AudioWorklet`（失败降级 `ScriptProcessor`），上行 20ms PCM 分片。  
9. 前端播放：维护按 `turn_id+seq` 的播放队列；Speaking 期间检测 RMS 超阈值触发 `interrupt` 并立刻 `stopPlayback()`。  
10. 打断策略：前端 RMS 触发 + 后端“Speaking 时收到有效新帧”双保险；默认阈值保守，先降误触发。  
11. 背压策略：上行队列满丢最旧；下行队列满则停止后续 TTS 分段并发 `error`（可恢复）。  
12. 观测策略：状态栏展示状态与错误；调试区展示 ASR/LLM 增量文本；关键时间点打点输出到前端调试面板。  

## 里程碑拆解
1. M1 骨架与配置：Go 项目初始化、配置加载、HTTP/WS 基础连通、静态页面可打开。  
2. M2 采集与 ASR：浏览器持续上行音频、后端识别并回传 `asr_partial/asr_final`。  
3. M3 LLM 与分段 TTS：识别 final 触发 LLM，LLM 增量分句触发多次 TTS，下行播放成功。  
4. M4 打断与恢复：Speaking 时打断可在 1s 内停播并进入新识别；Stop 在任意状态可回收。  
5. M5 稳定性与文档：补测试、修边界、更新项目说明与开发日志。  

## 测试用例与场景
1. 单元测试：配置解析、消息编解码、状态机转移、`turn_id` 绑定、分句器、取消幂等性。  
2. 集成测试（本地假 Provider）：WS 控制流、上行背压、下行分片配对、interrupt/stop 回收。  
3. 手工 E2E（真实 Provider）：正常多轮、长句、噪声、连续 3 次打断、Listening/Thinking/Speaking 各阶段 Stop。  
4. 回归重点：断网/接口失败后是否回到 Listening 或 Idle；是否存在 goroutine 泄漏和队列堆积。  
5. 验收指标：`t_play_start - t_u_end` 目标 < 2.5s（方向性指标）；打断停播 <= 1s。  

## 风险与应对
1. 官方协议字段差异：通过 Provider 适配层隔离，必要时仅改 `asr.go/tts.go/llm.go`。  
2. 浏览器音频兼容差异：保留 AudioWorklet 与 ScriptProcessor 双路径，优先可用性。  
3. 误打断：默认保守阈值 + 双触发校验 + 可配置阈值常量。  
4. 敏感信息泄露：禁止日志打印 token；仅加载 `config/configx.json`，模板文件一律脱敏。  
5. 本机 Git 环境差异：当前系统 `git` 受 Xcode License 影响，开发与验证使用可用的 Homebrew Git 路径。  

## 明确假设与默认值
1. 默认使用现有 `config/configx.json` 中的 `chat/asr_s/tts_s` 配置键名与 URL。  
2. 默认音频上行格式：PCM16LE、mono、16kHz、20ms 分片。  
3. 默认分句策略：中文标点 + 长度阈值（如 18~24 字）触发 TTS，避免“整段等完再说”。  
4. 默认队列大小：上行 64 帧、下行 24 段；超过即执行背压策略。  
5. 默认单会话模型：一个页面一个 WS，一个 WS 同时只允许一个 active turn。  
6. 默认监听地址：`127.0.0.1:18080`，不开放局域网。  
7. 若 `asr_s` 无 endpoint 信号：启用前端静音 `utterance_end`（RMS 阈值 + 800ms 静音窗）。
