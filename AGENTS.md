# AGENTS.md

## 1. 角色与最高原则
本文件是本项目内所有自动化开发与文档维护行为的最高指导原则。

执行优先级（从高到低）：
1. 真实性：所有描述必须可由代码、配置、Git 历史或可验证文件支撑。
2. 可追溯性：每次文档更新必须给出时间、范围、依据。
3. 一致性：文档内容必须与真实项目结构一致，禁止保留过期描述。
4. 可执行性：规则应能直接落地为操作步骤，不写空泛建议。
5. 技术中立性：本文档为纯粹的元准则与工作流规范，**绝对不能包含或涉及任何具体的开发技术栈要求或技术细节说明**（具体的技术栈、架构及代码限制应交由 `doc/_instruction.md`、`plan/` 及相关的 `.codex/skills/` 管理）。

## 2. 默认上下文加载规则
每个新会话开始时，按以下顺序加载上下文：
1. 优先加载 `doc/_instruction.md`（项目事实快照主文档）。
2. 若 `doc/_instruction.md` 不存在，则回退加载 `Instruction.md`，并在需要时迁移回 `doc/_instruction.md`。
3. 当任务涉及“历史变化、经验复盘、最近改动”时，额外加载 `doc/_devlog.md`。
4. 对关键结论，必须二次核验仓库真实文件（如 `find`、`rg --files`、`git log`、`git diff`）。

## 3. 文档定位与关系
1. `doc/_instruction.md`：项目当前状态的权威说明（结构、模块、术语、运行方式）。
2. `doc/_devlog.md`：按时间追加的开发记录（只增不改历史）。
3. `Instruction.md`：兼容历史文档。若存在，应与 `doc/_instruction.md` 保持一致，或在其中声明已迁移到 `doc/_instruction.md`。

## 4. 模式指令与权限模型（chat/plan/dev）
### 4.1 默认权限（每条消息）
1. 每个新对话默认处于 `chat` 权限：允许为理解、验证与分析问题执行完整的代码分析动作，包括项目扫描、读取本地可读文件、浏览网页、查看图片/PDF、查询 Git 历史与差异、运行分析工具与分析命令；允许在项目内创建、修改、删除用于分析的临时文件、缓存或中间产物，但不得修改、删除、重命名任何现有项目文件，不落地业务编码。
2. 对话过程中，若用户某条消息未明确呼唤 `plan` 或 `dev`，则该条消息仍按 `chat` 权限处理。
3. `plan` 在默认分析权限基础上，额外拥有对项目内任意 `*.md` 文档的完全控制权：可读取、创建、修改、重命名、删除任意路径下的 Markdown 文档，并可执行 `更新项目说明` 与 `更新开发日志` 等文档维护动作；对非 Markdown 的现有项目文件，仍不得修改、删除、重命名，但允许为分析任务生成和清理临时文件。需求说明、开发计划、开发结果等规划类文档默认写入 `plan/`（除非用户明确指定路径）。
4. `plan` 与 `dev` 默认是一条消息内的一次性授权，不会自动延续到下一条消息，除非用户再次明确呼唤或明确要求权限有持续性。

### 4.2 模式指令触发（禁止子串误触发）
模式指令必须是“独立标记”，避免误触发：
1. 指令词前后由空白或标点分隔（例如：`plan，请...`、`... dev：...`、`chat，...`）。
2. 若指令词紧随其后是 `.` `/` `\\` `-` `_` 或字母/数字，视为文件名/路径/标识符，不算模式指令（例如 `dev.go`、`plan.md`、`chatgpt` 不触发）。
3. 代码块、反引号包裹内容、URL 中出现的 `chat/plan/dev` 不视为模式指令。

### 4.3 同条消息多指令时的权限升级
同一条消息中出现多个模式指令时，不需要用户澄清，直接按“最高权限”执行：
1. 权限优先级：`dev` > `plan` > `chat`。
2. 若出现 `dev`：允许编辑项目内任意文件并执行必要验证；仍需对高风险/不可逆操作征求同意（例如大规模删除、破坏性重写、敏感凭据处理等）。
3. 若未出现 `dev` 但出现 `plan`：允许进行完整代码分析与核验（仓库扫描、本地文件读取、网页浏览、Git 查询、分析工具调用、分析命令执行等），允许在项目内生成和清理临时分析产物，并对任意路径下的 `*.md` 文档执行创建、修改、重命名、删除；不得对非 Markdown 的现有项目文件执行修改、删除、重命名，也不进行业务代码实现与工程化执行动作。

### 4.4 chat/plan 权限边界
1. `chat` 与 `plan` 都允许使用读取动作与代码分析动作，包括但不限于：目录扫描、文本检索、打开本地文档、查看图片/PDF、读取配置样例、查询 Git 历史与 diff、访问网页与在线文档、运行测试/静态检查/构建/脚本等分析型命令，只要目标是分析、验证、排查而不是落地实现。
2. `chat` 与 `plan` 都允许在项目内生成、修改、删除临时分析产物、缓存、中间文件或报告文件，前提是不覆盖、不改写、不删除任何现有项目文件。
3. `plan` 除了临时分析产物外，还可对项目内任意 `*.md` 文档执行正式写操作；该权限覆盖创建、修改、重命名、删除。规划类文档默认写入 `plan/`（除非用户明确指定路径）。
4. `chat` 与 `plan` 都禁止修改、删除、重命名现有项目代码、配置、文档、脚本、资源文件，以及执行提交代码、部署发布、破坏性清理等超出分析范围的动作。
5. 若确需生成文件，应优先使用可明确识别为临时用途的命名、路径或目录，避免与现有项目文件混淆。
6. 若某个命令、工具或目标路径是否会触达现有项目文件无法确认，默认按“未授权”处理，除非能够明确证明其安全且符合当前模式边界。

### 4.5 chat 输出风格（默认模式）
1. 精简优先：默认不给“面面俱到”的长清单；优先给结论与关键依据，必要时再展开。
2. 聚焦上下文：只围绕当前对话的主问题与约束回答，避免无关扩展与泛化建议。
3. 严谨准确：遇到不确定或关键事实，优先通过本地文件/Git/可复现输出核实；环境允许且必要时可做网络检索交叉验证；无法核实时明确标注假设与不确定性。
4. 深度思考：强调关键推理链路、关键权衡与决定性证据，而不是堆砌条目数量。

## 5. 触发词与动作映射（文档维护）
前提：写盘动作必须落在当前模式授权范围内。`chat` 仅可写入临时分析产物，不得改写现有项目文件；`plan` 可写入临时分析产物，并可正式写入任意 `*.md` 文档（规划类文档默认写入 `plan/`）；`dev` 可按需求写入项目内任意路径。在此前提下，出现以下意图时必须执行对应动作：
1. `更新开发日志` / `更新日志` / `记录最近改动` -> `plan` 或 `dev` 可更新 `doc/_devlog.md`（必须追加，不得改写历史记录）；若当前为 `chat`，只能先输出草案，不得落盘。
2. `更新项目说明` / `更新说明` / `同步项目文档` -> `plan` 或 `dev` 可更新 `doc/_instruction.md`（必要时同步 `Instruction.md`）；若当前为 `chat`，只能先输出草案，不得落盘。
3. 同一轮同时触发前两项任务时：若当前为 `dev`，先更新 `doc/_instruction.md`，再更新 `doc/_devlog.md`。
4. `制定开发计划` -> `plan` 或 `dev` 可产出并落盘开发计划文档；默认写入 `plan/<yymmdd>-<主题>-devplan.md`（不含时分秒，除非用户明确指定路径）。
5. `评估开发结果` -> `plan` 或 `dev` 可产出并落盘结果评估文档；默认写入 `plan/<yymmdd>-<主题>-devreport.md`（不含时分秒，除非用户明确指定路径）。

### 5.1 plan/ 文件命名约定（默认落盘）
1. 若用户明确指定输出路径或文件名，优先按用户指定。
2. 若用户未指定且默认落盘到 `plan/`，文件名必须以 `yymmdd-` 日期前缀开头（例如 `260312-`），不得额外包含小时/分钟/秒。
3. 文件类型后缀约定（均为 `.md`）：
   - 产品设计/功能需求/需求说明：以 `-prd` 结尾（例如 `plan/260312-<主题>-prd.md`）。
   - 开发计划：以 `-devplan` 结尾（例如 `plan/260312-<主题>-devplan.md`）。
   - 结果评估：以 `-devreport` 结尾（例如 `plan/260312-<主题>-devreport.md`）。
   - 纯分析报告：以 `-ana` 结尾（例如 `plan/260312-<主题>-ana.md`）。
4. `<主题>` 使用简短可读的关键词串（建议用 `-` 连接），能唯一表达本次文档主题即可；若无法确定主题，必须先向用户追问，而不是随意命名。

## 6. doc/_devlog.md 更新规范（重点）
### 6.1 更新目标
`doc/_devlog.md` 必须反映“从上一条日志到当前时刻”的真实开发增量，并沉淀成功与失败经验。

### 6.2 数据来源优先级
1. Git 证据：`git log --name-status`、`git diff`、提交信息。
2. 仓库差异：新增/删除/修改文件列表。
3. 对话事实：用户确认的决策、目标变化、问题反馈。
4. 若无 Git 历史可用，明确标注“基于文件变更与会话推断”。

### 6.3 写入规则
1. 只能追加新记录，不得改写历史记录内容。
2. 每条记录必须使用绝对时间（例如 `2026-03-03 18:30 CST`）。
3. 每条记录至少包含以下字段：
   - 时间范围：上一条日志时间 -> 当前时间
   - 主要变更：功能/修复/重构/文档/工程化（3-7 条高价值事项）
   - 变更文件与模块：列出关键路径，避免仅写笼统描述
   - 开发结果：成功项、失败项、阻塞项
   - 经验总结：本次有效做法、失败教训、后续规避策略
   - 下一步：明确可执行的后续动作
4. 若存在失败或回滚，必须记录原因、影响、恢复措施，不得省略。
5. 禁止记录“未发生”的修改或未经验证的结果。

### 6.4 推荐日志模板
```markdown
## [YYYY-MM-DD HH:MM TZ] <本次主题>
- 时间范围：<上次日志时间> -> <当前时间>
- 主要变更：
  - ...
- 关键文件/模块：
  - `path/to/file`
- 开发结果：
  - 成功：...
  - 失败/问题：...
- 经验与结论：
  - 有效实践：...
  - 失败教训：...
  - 后续规避：...
- 下一步：
  - ...
```

## 7. doc/_instruction.md 更新规范（重点）
### 7.1 更新目标
`doc/_instruction.md` 必须持续反映项目“当前真实状态”，并自动维护术语表与文件结构目录。

### 7.2 执行流程
1. 扫描真实结构：基于仓库根目录重新扫描目录与关键文件。
2. 对比旧文档：保留仍有效信息，删除过期模块和路径。
3. 合并近期开发事实：结合最近开发内容、成功/失败经验、实际变更结果更新对应章节。
4. 更新术语表：新增本轮出现的项目术语，清理废弃术语。
5. 更新目录树：自动刷新文件结构，确保与当前仓库一致。
6. 写入绝对更新时间，并标注“信息来源”（Git/文件扫描/会话确认）。

### 7.3 必备章节
`doc/_instruction.md` 至少包含：
1. 项目概览
2. 当前目录结构（关键层级树）
3. 核心模块职责
4. 开发与运行方式（仅写可验证内容）
5. 最近关键变更摘要（可来自最近 1-3 条 `doc/_devlog.md`）
6. 项目术语表（自动维护）
7. 待确认事项（无法验证的信息）
8. 文档更新时间（绝对日期时间）

### 7.4 项目术语表自动维护规则
1. 术语来源：
   - 对话中反复出现的项目专有词（设计或开发中经常使用的、特别约定含义的词），或用户特别说明的词。
2. 术语条目结构：
   - 术语
   - 定义（本项目语境）
   - 来源文件（至少一个路径）
   - 状态（active/deprecated）
3. 维护动作：
   - 新术语：自动新增。
   - 同义术语：合并并保留主术语。
   - 废弃术语：标记 `deprecated`，不直接删除历史痕迹。
4. 禁止加入通用空洞词（如“模块”“系统”）作为术语。

### 7.5 文件结构目录自动维护规则
1. 目录树必须来源于实时扫描，不可手写猜测。
2. 默认忽略噪音目录：`.git`、`node_modules`、`dist`、`build`、`.next`、`coverage`、临时目录。
3. 目录树展示关键层级（建议 2-4 层），并保留关键文件。
4. 每次更新需删除已不存在的路径，补充新增路径。
5. 若文档结构与真实结构冲突，以真实结构为准立即修正。
6. 目录树中**每个被展示的目录/文件**必须附带一句“简短用途说明”（同一行注释即可），且说明必须可由代码/配置/文件内容验证；若无法验证，必须标注为“待确认”而不是编造。

## 8. 一致性与安全约束
1. 若 `doc/_instruction.md` 与 `doc/_devlog.md` 冲突，以代码和 Git 证据为准，并在本次更新中修复冲突。
2. 禁止在文档中泄露密钥、令牌、密码；必要时使用掩码（如 `****`）。
3. 不确定的信息必须放入“待确认事项”，不得编造。

## 9. 完成标准（Definition of Done）
执行“更新项目说明/更新开发日志”后，必须同时满足：
1. 目标文档已更新或创建成功。
2. 内容与当前项目真实状态一致。
3. 包含绝对时间戳。
4. 可追溯到具体文件或提交依据。
5. 无虚构改动、无过期结构、无敏感信息明文扩散。
